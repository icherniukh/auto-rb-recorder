## Theories (labeled as theories, not facts)

### Theory A: macOS silent tap bug
`AudioHardwareCreateProcessTap` silently stops delivering Rekordbox's audio after an extended session. The IO proc continues running (buffers arrive on schedule) but their content is zeros. No error is raised. This is a known macOS behavior reported elsewhere. Evidence: clean 68-min run → abrupt zero transition with no device change or error. Against: could also be Rekordbox going idle.

### Theory B: CoreAudio state corruption accumulates over restarts
Each audiotee restart (SIGTERM or SIGKILL) tears down and recreates the aggregate device. After the first restart, the new tap only worked for 30 seconds before going to zeros again. All subsequent restarts produced immediate zeros. The degradation is progressive: 68 min → 30 sec → 0 sec. This matches fart-bug.md trigger #3.

### Theory C: Rekordbox thread exhaustion / audio engine idle
Rekordbox had ~300 threads after 9+ hours. If Rekordbox's CoreAudio output thread stalled or was starved, it could stop sending audio to the tap without any device change event. The tap would then deliver silence correctly. Cannot distinguish from Theory A without knowing if Rekordbox was actually playing at 00:26:21.

## CoreAudio Overload & "Fart" Bug Research (Update)

### Discovery Process
I analyzed `SUDOLOG.log` filtering for "Error" to understand why CoreAudio overloads were occurring. The logs were spammed with `HALS_OverloadMessage: Overload possibly due to HAL client proc exceeding io cycle budget.` and `HALC_ProxyIOContext::IOWorkLoop: skipping cycle due to overload`. This immediately points to the real-time IOProc thread taking too long to execute.

I then searched the `audiotee/Sources` directory for CoreAudio IOProc implementations (`AudioDeviceAddIOProc` / `AudioDeviceCreateIOProcID`) and the real-time callback (`processAudio`).
I noticed two massive violations of CoreAudio real-time safety rules:
1. **Swift ARC Traffic:** The callback invoked `Unmanaged<AudioRecorder>.fromOpaque(inClientData!).takeUnretainedValue()`, which introduces ARC (Automatic Reference Counting) `retain` and `release` calls. I verified this by compiling a minimal example and inspecting the emitted SIL (`swiftc -emit-sil`).
2. **Swift Exclusive Access Checks (Locks):** Inside the `AudioBuffer`'s `append` method, which is called from the RT thread, the code used `OSAtomicCompareAndSwap32Barrier(..., &writeIndex)` where `writeIndex` is a class property. Passing a class property as an `inout` pointer (`&`) forces Swift to emit `swift_beginAccess` and `swift_endAccess` calls to enforce memory exclusivity. These access checks take locks or modify thread-local state, causing priority inversion and blocking the RT thread.

### The Discoveries & Fixes
The combination of ARC overhead and Swift's dynamic access checks caused `audiotee` to miss its tight execution deadlines on the IOProc thread, leading to CoreAudio dropping the cycle (the "fart" bugs and silent drops).

To fix this, I made the following architectural changes to ensure strict real-time safety:
1. **Bypassed ARC:** Changed `Unmanaged...takeUnretainedValue()` to `Unmanaged..._withUnsafeGuaranteedRef { ... }`. This provides a temporary, unmanaged pointer to the object without triggering `retain`/`release` traffic.
2. **Eliminated Dynamic Access Checks:** Replaced the simple `Int32` class properties (`writeIndex`, `readIndex`, `droppedByteCount`) with manually allocated `UnsafeMutablePointer<Int32>`. By operating directly on raw pointers instead of class properties, Swift bypasses the dynamic exclusivity checks completely, allowing pure lock-free atomic operations.
3. **Inlined Method Dispatch:** Removed the `processAudio` method from `AudioRecorder` and inlined the buffer copy logic directly inside the C-callback block to avoid Swift's dynamic method dispatch overhead.

### CoreAudio Real-Time "Rules" Discovered
- **No Locks, No Blocking:** The IOProc thread runs with real-time priority. Any lock, mutex, or blocking operation (like `malloc`, `free`, `printf`, file I/O) can cause a deadline miss.
- **No ARC in Swift Callbacks:** Swift's Automatic Reference Counting (`retain`/`release`) is not lock-free and must never be used inside the IOProc thread. Use `Unmanaged._withUnsafeGuaranteedRef` to interact with objects safely.
- **No Dynamic Exclusivity Checks:** Using `inout` (`&`) on Swift class properties inside the RT thread triggers `swift_beginAccess`, which is not RT-safe. Use `UnsafeMutablePointer` for atomic operations to guarantee raw memory access without Swift runtime interference.
- **Avoid Dynamic Dispatch:** Inline critical path logic or use final/static methods to avoid the overhead of resolving Swift/Objective-C method tables during the RT callback.
